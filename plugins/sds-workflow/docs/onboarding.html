SDS 워크플로우 플러그인

Jira → Claude Code → GitLab MR 파이프라인 표준화

한 이슈를 처음부터 끝까지 따라가며 필요한 모든 컨텍스트 전환을 에이전트에 위임합니다. 사람은 판단·검토·승인에 집중합니다.

2026-04 · Okestro SDS 팀 · Claude Code Plugin v0.1.0

왜 도입했는가

현재의 마찰 — 하루에도 수십 번의 컨텍스트 전환

👤개발자
⇄ Jira ⇄ IDE ⇄ Terminal ⇄ GitLab ⇄ Confluence

브랜치 prefix · commit type · Jira transition · MR 템플릿 — 전부 손으로. 누락 1건 → 리뷰 지연 또는 규약 이탈.

플러그인 도입 후 — 단일 슬래시 커맨드가 모든 컨텍스트를 흡수

👤개발자
→ /pick/ship/land/recap → Jira IDE Terminal GitLab Confluence

컨텍스트 전환은 에이전트 내부에서 일어난다. 사람은 한 창에서 판단·승인만.

집기 → 출하 → 착륙 → 보고 의 4단계 파이프라인으로 압축. 각 단계는 슬래시 커맨드 1회로 자동화.

시작하기

구성: 1·2·3 은 PC 당 1회, 4 는 저장소마다 1회, 5 는 이슈마다 반복. marketplace add·plugin install 은 각 사용자의 ~/.claude/ 에 기록되므로 동료가 자기 PC 에서 한 번씩 직접 실행해야 합니다.

1. 사전 도구 설치·인증

  • acli — Atlassian Jira CLI
  • glab — GitLab CLI
  • 각 커맨드 실행 중 미설치가 감지되면 설치 안내가 자동으로 뜹니다.

2. 마켓플레이스 등록 (사용자 계정당 1회)

# A. 로컬 clone 경로 (ceph-web-ui 작업자 권장)
/plugin marketplace add /path/to/remote-ceph-admin

# B. 또는 git URL + dev 브랜치 핀 (저장소를 clone 하지 않을 경우)
/plugin marketplace add http://gitlab.prd.console.trombone.okestro.cloud/SDS306/remote-ceph-admin.git#dev

B 의 #dev 핀은 기본 브랜치 prd 에 .claude-plugin/marketplace.json 이 머지될 때까지 한시적으로 필요합니다. 머지 이후에는 핀 제거 가능.

3. 플러그인 설치 (사용자 계정당 1회)

/plugin install sds-workflow@remote-ceph-admin

4. 플러그인 reload (첫 설치 · 업데이트 후 필수)

# Claude Code 세션 안에서 슬래시 커맨드 인덱스를 새로고침
/reload-plugins

# 반영이 안 되면 Claude Code 를 완전히 재시작 (새 탭 / 프로세스 재실행)

설치 직후에는 현재 세션이 아직 구버전 커맨드 인덱스를 쥐고 있습니다. /reload-plugins 를 실행하면 플러그인 목록·커맨드가 즉시 재로드되어 /sds-workflow:pick 등이 자동완성에 뜹니다. 플러그인 버전 업데이트 시(/plugin marketplace update) 도 동일하게 /reload-plugins 를 한 번 실행하면 됩니다.

5. 새 저장소 부팅 (저장소마다 1회)

cd <project-root>
/sds-workflow:init

/sds-workflow:init 이 .team-workflow/ 스캐폴드를 생성하고 project_key·jira.base_url·confluence.base_url·confluence.space_key 를 대화형으로 받습니다. 이후 CONVENTIONS.md 를 저장소 현황에 맞춰 1회 정리하면 운영 준비 완료.

6. 첫 이슈 시작

/sds-workflow:pick CDS-XXXX    # 또는 막히면 /sds-workflow:where 로 현재 상태 확인
커맨드 접두사 — Claude Code 플러그인은 기본적으로 <plugin-name>:<command> 형태로 호출합니다. 이 문서에서도 /sds-workflow:pick 등 풀 네임을 기본 표기로 씁니다. 충돌이 없을 때는 Claude Code 가 단축 /pick 으로 매칭해 주기도 하지만, 다른 플러그인과 이름이 겹치면 모호해지므로 접두사 포함이 안전합니다.

파이프라인 개요

STAGE 1
pick
이슈 가져오기 · 컨텍스트 수집 · 플랜
→
STAGE 2
ship
검증 · 커밋 · push · MR 등록
→
STAGE 3
land
MR 머지 후 Jira 전환
→
STAGE 4
recap
결과 코멘트 · Confluence 보고

보조 커맨드

  • /sds-workflow:where — 현재 상태 감지 후 다음 액션 안내
  • /sds-workflow:draft — 자유 프롬프트 → 새 Jira 이슈 초안 (5섹션 구조화 + 중복 탐지)
  • /sds-workflow:autopilot — pick → 구현 → ship 자율 운행 (다중 이슈 전달 시 worktree 병렬)
  • /sds-workflow:tune — 운영 중 발견한 피드백을 워크플로우에 구조화해 반영
  • /sds-workflow:init — 새 저장소 부팅 (스캐폴드 생성)

커맨드 치트시트

커맨드입력산출물
/sds-workflow:pick CDS-XXXXJira 이슈 키브랜치 · Jira 전환 · 플랜 모드 진입
/sds-workflow:ship작업 완료 상태검증 · 커밋 · push · MR · Jira 코멘트
/sds-workflow:land CDS-XXXX머지된 이슈 키Jira RESOLVE (로컬 브랜치는 보존)
/sds-workflow:recap CDS-XXXX [--confluence]완료 이슈 키Jira 결과 코멘트 (+ Confluence 페이지)
/sds-workflow:where-현재 상태 기준 다음 액션 안내
/sds-workflow:draft "자유 프롬프트"아이디어 텍스트신규 Jira 이슈 초안 (5섹션)
/sds-workflow:autopilot CDS-XXXX [CDS-YYYY ...] [--stop-at <phase>] [--ralph]Jira 이슈 키 1개 또는 여러 개 (공백 구분)단일: pick → 구현 → ship 자동 · 다중: worktree 격리 + subagent 병렬 (--ralph 자동 강제)
/sds-workflow:tune "피드백"자유 형식 피드백수정 제안 + tune-log 기록
/sds-workflow:init새 저장소.team-workflow/ 스캐폴드
접두사 — 이 문서는 풀 네임 /sds-workflow:<command> 를 기본 표기로 씁니다. Claude Code 가 단축 /pick 로 매칭해 주기도 하지만, 다른 플러그인과 이름이 겹치면 모호하므로 접두사 포함이 안전합니다.

밤사이 비행 — 사람은 감독에 집중

하루 한 이슈 수동 처리는 기존 개발과 다를 바 없습니다. 진짜 변화는 병렬 + 야간 자동화 에서 일어납니다.

🌆 EVENING · 퇴근 직전

비행 계획 수립

  • 여러 이슈 키를 한 줄로 전달 → /sds-workflow:autopilot 이 worktree 병렬 모드로 분기
  • 또는 이슈별 탭 분리, 또는 Claude Code 스케줄러로 야간 예약
  • 사람은 "무엇을 시킬지" 결정만
🌙 NIGHT · 무인 운항

에이전트가 처리

  • 이슈 읽기 → 브랜치 생성 → Jira In Progress
  • 컨텍스트 수집 → 플랜 초안 → 구현
  • 린트·타입·테스트 → 커밋 → push
  • GitLab MR 생성 → Jira 에 링크 코멘트
☀️ MORNING · 사람의 시간

감독과 창의

  • 쌓인 MR 리뷰·승인 (핵심 판단)
  • /sds-workflow:draft 로 다음 이슈 정의
  • 기획·아키텍처·회의
  • 머지된 건은 /sds-workflow:land → /sds-workflow:recap

예시 — 퇴근 전 30초 셋업

# 패턴 1 — 한 줄 다중 이슈 (worktree 자동 격리 + --ralph 자동 강제)
/sds-workflow:autopilot CDS-2150 CDS-2151 CDS-2152

# 패턴 2 — 탭/세션 분리 (한 이슈당 한 대화, 플랜 승인 게이트 유지)
/sds-workflow:autopilot CDS-2150      # 탭 A
/sds-workflow:autopilot CDS-2151      # 탭 B

# 패턴 3 — 야간 예약 실행
/schedule "weekdays 22:00" /sds-workflow:autopilot CDS-2152
다중 이슈 모드는 비슷한 패턴의 trivial ~ medium batch 에 한정 권장. 도메인이 교차하거나 상호 의존하는 이슈는 단일 모드 순차 실행이 충돌·재작업이 적습니다. 승인 게이트가 자동 스킵(--ralph)되므로 리뷰어 부담이 증가합니다.
핵심 전환 — 반복 구현 노동은 오토파일럿에게 위임. 사람은 AI 가 못 하는 일(기획·감독·판단·리뷰)에 집중. 산출물은 아침에 MR 큐로 도착.

권장 퍼미션 — 한 번 허용해 두면 흐름이 끊기지 않습니다

/sds-workflow:* 파이프라인은 외부 CLI (acli·glab·git·pnpm) 를 Bash 로 호출합니다. 처음 만난 명령마다 Claude Code 가 퍼미션 프롬프트를 띄우므로, 자주 쓰는 패턴을 미리 허용 해 두면 야간·다중 모드 자율 운행 중에 멈추지 않습니다.

필요 퍼미션 역추적

허용 패턴왜 필요사용 커맨드
Bash(acli:*)Atlassian Jira CLI — 이슈 조회·전환·코멘트·생성pick · ship · land · recap · draft · autopilot
Bash(glab:*)GitLab CLI — MR 생성ship
Bash(git:*)브랜치·커밋·push·diff·status·worktree모두
Bash(pnpm:*)정적 검증 (lint·type-check·test)ship (Phase 1)
Bash(command -v:*)도구 설치 여부 감지acli 쓰는 모든 커맨드 Phase 0
Bash(bash:*) 또는 Bash(${CLAUDE_PLUGIN_ROOT}/scripts/*.sh:*)플러그인 스크립트 실행 (create-mr.sh·jira-comment.sh)ship
Bash(mkdir:*)·Bash(cp:*)·Bash(grep:*)·Bash(printf:*)seed 복사 + .gitignore 패치init

사전 허용 — 두 가지 방법

A. 사용자 계정 전체 (모든 저장소에서 유효) — ~/.claude/settings.json 에 아래를 추가:

{
  "permissions": {
    "allow": [
      "Bash(acli:*)",
      "Bash(glab:*)",
      "Bash(git:*)",
      "Bash(pnpm:*)",
      "Bash(command -v:*)",
      "Bash(bash:*)",
      "Bash(mkdir:*)",
      "Bash(cp:*)",
      "Bash(grep:*)",
      "Bash(printf:*)"
    ]
  }
}

B. 프로젝트 한정 — 저장소 루트 .claude/settings.json 에 같은 블록을 넣으면 그 저장소에서만 유효합니다. 팀원이 공유하는 기준치를 한 곳에서 관리하고 싶을 때 유용.

보안 메모 — Bash(acli:*) 같은 와일드카드는 해당 CLI 전체를 허용합니다. Jira/GitLab 자격증명이 설치된 PC 에서만 쓰고, Bash(rm:*) 같은 파괴적 패턴은 절대 일괄 허용하지 않습니다.

피드백은 /sds-workflow:tune 으로

"이 커맨드가 이렇게 동작했으면 좋겠다" — 구두·Slack 에 흘리지 말고 시스템에 녹입니다.

/sds-workflow:tune "pick 단계에서 기능 개요 브리핑을 더 상세히 해줬으면 좋겠다"

자동 수행

  • 카테고리 판정 — command-behavior / template / config / config-local / design-principle / convention / registry
  • 대상 파일 후보 탐색 + 레지스트리 기반 일관성 전파 대상 자동 포함
  • Diff 초안 작성
  • E2E 영향·리스크 평가

사람 개입

  • Diff 검토·승인
  • 즉시 적용 / 대기 (deferred) / 폐기 (rejected) 선택
  • 커밋 메시지 승인
폐기한 피드백도 tune-log.md 에 기록됩니다. 같은 주제가 반복 제기되면 의사결정 근거로 활용.

├── README.md

플러그인 repo 가 소유. 수정은 /tune + PR.

저장소 소유 데이터

.team-workflow/
├── workflow.yml              ← project_key 등 override
├── CONVENTIONS.md            ← 저장소 비타협 규칙
└── tune-log.md               ← 피드백 이력

저장소마다 고유. /sds-workflow:init 이 seed 복사로 생성.

안전 장치 · 원칙

  • 파괴적 작업 금지 — 파일 삭제·force push·브랜치 강제 삭제 없음
  • Amend 금지 — 커밋 후 추가 커밋으로 이력 보존
  • 로컬 브랜치 보존 — /sds-workflow:land 는 Jira 전환만 수행. 재참조·cherry-pick 여지를 남김
  • E2E 영향 자동 판정 — 워크플로우 변경이 파일럿(pick→ship→land→recap) 을 깨뜨릴 가능성 있으면 needs-e2e 로 보류
  • SPEC.md 확정 사항 변경 시 추가 확인 — 파일럿 철학은 쉽게 바뀌면 안 됨
  • 사람 개입 지점 — 플랜 승인 · Diff 검토 · 커밋 메시지 승인 · 그레이존 결정
  • 제품 코드 수정은 대상 아님 — /sds-workflow:tune 은 워크플로우 시스템만. 제품 코드는 /sds-workflow:pick 또는 /sds-workflow:draft
