빌드 의사결정 4가지

"어디서 어떻게 패키징 할까" 의 결정 과정

1
Marketplace + Plugin 같은 repo 통합 (분리 X)
단일 plugin 1-2개라면 marketplace.json 과 plugins/ 가 같은 repo 동거가 단순. 분리는 plugin 여러 개 큐레이션할 때만 가치 발생.
2
민감정보는 consumer repo 의 .team-workflow/ 에
Jira project key, GitLab URL 같은 환경 의존 값은 plugin 안에 박지 않음. .team-workflow/workflow.yml 가 deep-merge 대상. plugin 은 placeholder 만 갖고 generic 유지.
3
2개 plugin 으로 분리 — sds-workflow + weekly-report
기능 성격이 다름 (개발 워크플로우 vs 보고 자동화) + 의존 도구 다름 (acli/glab vs Confluence Python lib). 별 plugin 으로 두면 사용자가 필요한 것만 install.
4
version 필드 제거 → commit SHA 기반 auto-update
활발한 개발 단계엔 매 push 마다 version bump 부담. plugin.json 에서 version 빼면 Claude Code 가 commit SHA 를 자동으로 version 으로 사용 → 사용자의 auto-update 가 변경 감지. stable 릴리스 시점에 다시 추가.
